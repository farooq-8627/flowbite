## 🔧 New-pipeline auto-pin made SSOT-driven + self-healing — system fields appear reliably on every new Default stage — 2026-06-10

User reported screenshot evidence that the previous-ship's `f.system === true` filter was now blocking ALL field auto-pins on a newly-created pipeline ("Client" pipeline → 0 fields on Default stage, was producing every-field-attached before). Root cause: orgs onboarded via the production path (`updateOrgIndustry` → `setupWorkspaceFromTemplate`) **never get the BUILT_IN system field seed run** — `setupWorkspaceFromTemplate` only seeds the curated platform-template's `fieldDefinitions` slot (industry-specific overlays like `property_type`, `bedrooms`), not the universal BUILT_IN_DEAL_FIELDS list. The lazy `ensureFields` `useEffect` in `PipelinesGroup.tsx` plugs the gap when the page loads, but it's async fire-and-forget — a quick "+ New pipeline" click can race ahead. Per-row `f.system` flag is therefore unreliable as the source of truth for "is this a system field".

**Fix.** Made the auto-pin block in `convex/crm/fields/pipelines/mutations.ts::createImpl` SSOT-driven against `BUILT_IN_DEAL_FIELDS` from `convex/orgs/templates/fields.ts` (now `export`ed). The block now:

1. **Inserts on demand.** For each of the 6 baseline system deal fields (dealCode, title, value, currentStageId, assignedTo, tags) that's missing from the org's `fieldDefinitions`, the field is inserted with `system: true`, `protected: <seed.protected ?? false>`, `showInStages: [newDefaultStage.id]` — race-condition-proof, no longer depends on `ensureFields` having completed.
2. **Self-heals stale flags.** For each existing row matching a BUILT_IN seed name, patches `system` and `protected` to the SSOT values when they drift (handles orgs whose rows were inserted via the curated-template path with `system: false`).
3. **Pins idempotently.** Extends `showInStages` with the new Default stage id only when not already present.
4. **Skips custom fields entirely.** Anything not in `BUILT_IN_DEAL_FIELDS` (industry custom fields like `property_type`, admin-added fields) is left untouched — same intent as the previous ship, just driven by name not flag.

The previous `getOtherDefaultStageIds` helper + the `existing.length === 0 / existing.every(otherDefaults.has)` branching are gone (per AGENTS.md §1.1 side-by-side cleanup — no dead-code parallel paths). The new approach is simpler, fully covers the case the previous logic missed (BUILT_IN fields not yet seeded), and keeps the behaviour the user asked for ("only system fields on a new pipeline's Default stage").

**Files**: `convex/orgs/templates/fields.ts` (`export` added to `BUILT_IN_DEAL_FIELDS`), `convex/crm/fields/pipelines/mutations.ts` (rewritten auto-pin block in `createImpl` + import of `BUILT_IN_DEAL_FIELDS` + removal of dead `getOtherDefaultStageIds` helper).

**No schema migration required** — the auto-pin is a runtime-side-effect on every pipeline-create call, which both inserts missing system field rows AND self-heals stale flags lazily as orgs create new pipelines. Verified: typecheck 0 · biome 0/0 (1132 files) · `pnpm test` 1278 pass / 1 skipped (75 files) · `pnpm exec vitest run` 215 pass (16 files) · `pnpm build` ✅.

## 🗑️ Settings → Pipelines: deletion (incl. default) when deal-count = 0 + new-pipeline auto-pin restricted to system fields — 2026-06-10

User reported two pipeline-settings gaps: (a) no UI affordance to delete pipelines (including the default seeded by industry templates) — admins were stuck with whatever templates shipped; (b) creating a new pipeline auto-pinned every existing custom field onto the new pipeline's Default stage, so a real-estate org with 13 industry-template fields saw all 13 surface on every additional pipeline they created. Both shipped end-to-end. Verified: `pnpm typecheck` 0 · `pnpm exec biome check .` 0 errors / 0 warnings (1132 files) · `pnpm test` 1278 pass / 1 skipped (75 files) · `pnpm exec vitest run` 215 pass (16 files) · `pnpm build` ✅ on every route. **No schema migration required** — both changes are runtime-behaviour only.

**(1) Default pipelines are deletable when empty.** `convex/crm/fields/pipelines/mutations.ts::deletePipelineImpl` rewritten: dropped the unconditional `if (pipeline.isDefault) throw` refusal; replaced the per-stage deal scan (which used `by_org_and_stage` + ran one query per stage) with a single `by_org_and_pipeline` index hit that catches deals in EVERY stage including soft-deleted rows. Refusal logic now: any deal at all (including `deletedAt !== undefined`) → `PIPELINE_HAS_DEALS` with friendly copy `"Cannot delete a pipeline that has deals. Move or delete the deals first."`. After a successful delete: when the deleted pipeline was the default AND another deal pipeline exists for the org, the oldest remaining one (by `_creationTime`) is auto-promoted to `isDefault: true` so the next deal-create round-trip still resolves a default. When zero deal pipelines remain, the org is left empty (matches the user's "delete what we don't need" framing — they create a new one when ready). New `pipeline_deleted` activity-log row carries `metadata: { wasDefault, entityType }`.

**(2) New pipeline auto-pin restricted to `system === true` fields.** Same file's `createImpl` deal-fields loop now skips every fieldDefinitions row where `f.system !== true`. Net effect: the seeded baseline (`displayName`, `email`, `phone`, `status`, `source`, etc.) still surfaces on the new pipeline's Default stage; admin-added or industry-template custom fields stay where they were and the owner pins them via the Stage fields tab when they want them on the new pipeline. The `existing.length === 0 → pin` and `existing.every(otherDefaults) → extend` branches remain (now scoped to system fields only). Code comment at the loop site documents the rule + cites this entry.

**(3) New `countByPipelines` orgQuery + UI delete affordance.** `convex/crm/fields/pipelines/queries.ts` exports `countByPipelines({ orgId })` returning `Record<pipelineId, number>` across the org's deal pipelines (bounded `take(500)` per pipeline — UI displays "500+" past that). `core/platform/settings/components/groups/pipelines/PipelineEditor.tsx` gained `dealCount` + `onDeleted` props; the editor header now shows a kebab `<DropdownMenu>` with a single "Delete pipeline" item, gated on `dealCount === 0` (disabled with `<Tooltip>` explaining the deal blocker). ## 🛑 `bulk_create_entities` first-try success + post-success retry/duplicate loop killed (Bug A + Bug B + stuck-on-yellow `getMessageContent`) — 2026-06-07

User reported `bulk_create_entities` failing on the first 2 attempts of a "create 5 leads" request (`status:"new"` + placeholder `assignedTo:"member-1"` rejected), then after the third attempt finally succeeded the AI looped and created 24 rows (8 Alice / 5 Carla duplicates) while stacking 2FA prompts forever. Three coupled fixes shipped + 3 regression-pin tests + 1 documentation lock. Verified: `pnpm typecheck` 0 errors · `pnpm exec biome check .` 0 errors / 0 warnings (1132 files) · `pnpm test` 75 files, **1278 passed / 1 skipped (was 1272 → 1278 = +3 new pins + 3 from elsewhere this session)**.

**(Bug A) First-call validator pre-filter + Convex-ID shape gate.** `convex/crm/shared/bulk/capabilities.ts` gained `ENTITY_CREATE_ARG_ALLOWLIST: Record<EntityKey, ReadonlySet<string>>` (mirrors every `*ForAI` mutation's accepted args — lead/contact/deal/company) and `CONVEX_ID_PATTERN = /^[a-z0-9]{25,40}$/`. Inside `bulkCreateEntities.run`'s per-row loop, immediately after `partitionRowKeys(...)` and BEFORE `buildCreateArgs(...)`, two new gates: (a) any key in `partition.columnArgs` not in the entity's allowlist is `delete`d and surfaced via `warnings[]` with reason `Dropped N keys not accepted by ${entityType}.create: ${keys}. Set these via update_entity after the row is created.`; (b) `partition.columnArgs.assignedTo`, when a string failing the Convex-ID regex, is dropped with reason `Dropped placeholder assignedTo "${value}" — not a Convex user _id. Row created unassigned.`. Root cause: the dispatcher in `dynamicFieldDispatch.ts` correctly routes ANY `storage:"column"` field def into `columnArgs`, but `leads.status` IS a column on the schema yet the `lead.createForAI` validator does NOT accept it (status is hardcoded to "new" inside `createImpl`). Without the filter, Convex's runtime validator rejected the WHOLE row with `ArgumentValidationError: extra field 'status'` BEFORE the handler ran, and since the AI emitted the same shape per row, the entire batch failed. Same path with `assignedTo:"member-1"` — small models (GPT-OSS 120B Free, Llama, Gemini Flash) hallucinate placeholder ID strings when they don't have a real users `_id` in context; the `v.id("users")` validator killed the whole row.

**(Bug B) Clean OK headline + explicit STOP playbook line.** Two same-file edits: (a) the `bulk_create_entities` `ok`-branch headline drops the `(${N} rows had unrecognised keys — see warnings)` suffix (warnings still land in `data.warnings[]` for the model + UI to inspect); (b) the `bulk` group `playbook` opens with a NEW paragraph: `**STOP rule (locked 2026-06-07):** When a bulk tool returns 'status: "ok"', the operation is COMPLETE — even if the envelope's data.warnings[] lists rows with unrecognised keys. Warnings mean "the row was created and these extra keys were dropped" — they are NOT a failure signal and do NOT require a retry. ... NEVER call bulk_create_entities (or any bulk tool) a second time inside the same turn after an 'ok' — that path duplicates rows and burns 2FA tokens. Re-call ONLY on needs_repair, infra_retry, or after the user explicitly asks for another batch.` Root cause: the headline suffix made `status:"ok"` read like a partial-success to small models, which then re-invoked the tool to "fix the warnings" — every retry triggered a fresh `needs_step_up` (each token is single-use), and each user-approved retry created another duplicate row (`createImpl`'s email-dedup throws `ConvexError(DUPLICATE)` ONLY when `args.email` is present; the AI's retries sometimes dropped the email field, slipping past dedup).

**(Bug #1 stuck-on-yellow) `_readForTest` regression fix.** Already in code under a `locked 2026-06-07` comment but not yet recorded here. `convex/ai/messages.ts:500` `getMessageContent` (internalQuery, takes `messageId: v.id("aiMessages")`, returns `{content, thinkingState, conversationId, orgId} | null`) restored after a 2026-06-06 V1-style-inline-approval ship referenced `ai/messages:_readForTest` — a path that never existed in `messages.ts` (the `_readForTest` that does exist is in `convex/aiStepUp.ts`, an internalMutation that takes `tokenId`). **Regression pins added** (`convex/crm/shared/bulk/capabilities.test.ts` +3 tests at the bottom): (a) the input zod schema accepts a row with `status:"new"` + `assignedTo:"member-1"` (the runner is what filters them, not zod), (b) `getGroup("bulk").playbook` contains both `"STOP rule"` and a `do not retry|never call` directive plus references `` `ok` ``, (c) `bulk_create_entities.drive.onSuccess` does NOT contain `retry|try again` wording.
